# 《不完美的小店》TODO 路线图

## 当前最高优先级

先验证并锁定基建，不继续堆复杂玩法。

最先要做：

1. 用 Unity `2022.3.62f3` 打开项目。
2. 打开 `Assets/_Project/Scenes/S_Bootstrap.unity`。
3. 点击 Play，确认没有编译错误、脚本丢失或资源引用丢失。
4. 手动跑通至少 1 天流程：查看订单 → 选择订单 → 修补 → 夜晚结算 → 进入下一天。
5. 如果有错误，优先修编译、asmdef、场景引用和 ScriptableObject 数据，不新增玩法。

## Phase 0：基建锁定

目标：让项目结构稳定，后续开发知道文件放哪、模块怎么依赖、数据怎么配。

- [ ] Unity 打开项目无编译错误。
- [ ] `S_Bootstrap.unity` 无脚本丢失。
- [ ] `Assets/_Project/` 作为唯一业务资产主目录。
- [ ] `Core`、`Data`、`Gameplay`、`Flow`、`UI`、`Save`、`Utils` asmdef 依赖方向稳定。
- [ ] 中文文档、脚本、Unity 文本资产统一使用 UTF-8。
- [ ] 样例材料、顾客、订单、初始配置资产引用完整。
- [ ] README、PRD、MVP、架构、路线图、开发流程文档链接可用。

验收标准：

- 能打开正式入口场景并进入 Play。
- 无编译错误、无 Missing Script、无明显乱码。
- 不需要改代码即可看到原型 UI。

## Phase 1：MVP 闭环

目标：跑通最小可玩经营循环。

- [ ] 第 1 天加载 3 个订单。
- [ ] 玩家可以选择订单并查看详情。
- [ ] 修补订单时校验材料和能量。
- [ ] 成功修补后扣材料、扣能量、加金币、记录反馈。
- [ ] 夜晚结算显示完成订单数、当日收入、剩余资源和反馈摘要。
- [ ] 可以进入下一天。
- [ ] 第 3 天后显示原型完成提示。

验收标准：

- 玩家能从第 1 天连续玩到第 3 天。
- 至少能体验一次材料不足或能量不足提示。
- 不需要正式美术也能理解核心流程。

## Phase 2：Alpha 玩法层

目标：让“修补选择”和“顾客偏好”开始产生策略差异。

- [ ] 加入修补方式：完美复原、保留痕迹、创意再造。
- [ ] 顾客类型影响反馈和奖励倾向。
- [ ] 真实度或口碑开始被订单结果影响。
- [ ] 加入基础材料购买。
- [ ] 加入基础店铺升级，占位也可以。
- [ ] 把程序化 UI 逐步替换为可维护 Prefab。

验收标准：

- 同一个订单可以因为修补方式不同产生不同反馈。
- 玩家开始需要考虑“赚钱、资源、真实感”的取舍。

## Phase 3：Beta 表现层

目标：增强叙事氛围和可持续游玩体验。

- [ ] 加入手机入口。
- [ ] 加入日记或朋友圈反馈。
- [ ] 加入新手引导。
- [ ] 增加更多反馈文本和顾客备注。
- [ ] 加入存档基础能力。
- [ ] 做第一轮数值平衡。

验收标准：

- 玩家能理解每个系统入口。
- 每天经营后能看到情绪反馈和轻叙事回响。
- 关闭重开后基础进度可恢复。

## Phase 4：内容扩展

目标：扩充游戏内容，不破坏 MVP 闭环。

- [ ] 增加顾客故事线。
- [ ] 增加特殊订单。
- [ ] 增加店铺装饰变化。
- [ ] 增加长期成长目标。
- [ ] 探索多结局方向。

验收标准：

- 内容扩展围绕订单闭环增加，不重写核心架构。
- 玩家能感到小店逐渐变得“像自己”。
