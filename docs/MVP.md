# 《不完美的小店》MVP 范围说明

## 1. MVP 目标

MVP 的目标是做出第一版可玩的经营闭环，让玩家能连续体验数天小店经营，并感受到核心选择：

- 今天接哪个订单？
- 材料和能量够不够？
- 修补后能赚多少钱？
- 夜晚要不要先补材料，还是攒钱等后续升级？
- 顾客反馈会如何影响后续经营？

MVP 不追求完整剧情、美术完成度或复杂系统，只验证“订单选择 → 修补 → 交付 → 结算 → 下一天”的基础乐趣。

## 2. 必须实现的玩家流程

1. 进入一天开始界面
2. 查看当天订单列表
3. 选择一个或多个订单
4. 检查所需材料与能量
5. 执行修补
6. 交付给顾客
7. 获得收入与简短反馈
8. 夜晚结算当日收入、能量和材料变化，同时购买新的材料
9. 进入下一天

## 3. MVP 核心系统

### 订单系统

每个订单至少包含：

- 订单名称
- 物品类型
- 损坏程度
- 所需材料
- 能量消耗
- 报酬
- 顾客备注
- 顾客反馈文本

MVP 可先准备少量固定订单，不需要动态生成复杂订单池。

### 修补系统

MVP 只需要支持基础修补行为：

- 检查材料是否足够
- 检查能量是否足够
- 扣除材料
- 扣除能量
- 标记订单完成
- 发放收入和反馈

修补方式可先只做一种默认方式。完美复原、保留痕迹、创意再造可作为 Alpha 扩展。

### 资源系统

MVP 保留三个核心资源：

- **收入**：完成订单获得，用于表达经营进展。
- **能量**：限制每日订单数量。
- **材料**：限制可完成订单类型。

真实度可以先作为显示值或预留字段，不要求在 MVP 里承担复杂分支。

### 结算系统

每天结束时显示：

- 当日完成订单数
- 当日收入
- 剩余能量
- 剩余材料
- 顾客反馈摘要
- 可购买材料入口

结算后进入下一天，并恢复部分或全部能量。

### 基础 UI

MVP UI 只需要支撑流程清楚：

- 订单列表
- 订单详情
- 资源栏
- 修补按钮
- 结果反馈
- 夜晚结算面板
- 简化材料购买面板

可以使用占位图、简单按钮和文本，不要求最终视觉表现。

## 4. 暂缓内容

以下内容不进入 MVP：

- 朋友圈
- 日记系统
- 店铺评价详情页
- 复杂店铺升级
- 房租、每日杂费和复杂长期经济压力
- 深度顾客类型分层
- 完美复原 / 保留痕迹 / 创意再造三选一修补策略
- 长剧情、多结局或大规模文本内容
- 复杂口碑分流和订单池权重
- 商店装饰、美术精修和完整音效

这些内容应在核心循环可玩后逐步加入。

## 5. 推荐首版数据

MVP 可先使用非常小的数据量：

- 3 天经营流程
- 每天 3 个可选订单
- 5 种基础材料：胶水、布料、木料、旧零件、颜料
- 1 个简化材料购买入口
- 6 到 9 条顾客反馈文本
- 1 个夜晚结算界面

这样可以快速验证玩法，而不会被内容量拖住。

## 6. 验收标准

MVP 完成时，应满足：

- 玩家可以从第一天玩到第三天，不需要开发者手动干预。
- 玩家能看懂订单需要什么、会消耗什么、能获得什么。
- 材料不足或能量不足时，游戏能明确提示不能修补。
- 完成订单后，收入、能量和材料会正确变化。
- 夜晚结算能清楚展示当日结果，支持购买材料，并能进入下一天。
- 即使使用占位美术，也能感受到“小店修补经营”的主题。

## 7. 后续扩展入口

MVP 代码和数据结构应为后续扩展预留空间：

- 订单可增加顾客类型和口碑标签。
- 修补可增加多种方式。
- 结算可增加真实度和店铺评价。
- 夜晚阶段可扩展为手机、日记和朋友圈。
- 收入可用于材料购买和店铺升级。
