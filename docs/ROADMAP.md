# 《不完美的小店》开发路线图

## 1. 当前优先级

当前项目处于早期 Unity 空工程阶段，优先级是基建先行：

1. 固定工程目录和脚本分层。
2. 定义 ScriptableObject 数据模板。
3. 建立空的主流程入口。
4. 再开始 MVP 订单、修补、结算闭环。

## 2. Phase 0：基建

目标：让项目有稳定骨架，后续开发不再纠结“东西放哪、怎么接”。

必须完成：

- 建立 `Assets/_Project/` 目录结构。
- 新增架构、路线图、开发流程文档。
- 建立 `Core`、`Data`、`Flow`、`Gameplay`、`UI`、`Save`、`Utils` 脚本模块目录。
- 创建材料、订单、顾客、修补方式、初始配置的 ScriptableObject 模板。
- 创建 `GamePhase` 主流程阶段枚举。
- 创建空的 Bootstrap 场景入口。

验收标准：

- Unity 能打开项目。
- `_Project` 目录完整。
- 数据模板能编译。
- README 能链接到所有核心文档。

## 3. Phase 1：MVP 闭环

目标：做出第一版可玩的“订单 → 修补 → 结算 → 下一天”。

必须完成：

- 显示当天订单列表。
- 支持选择订单。
- 检查材料与能量是否足够。
- 扣除材料和能量。
- 完成订单并发放收入。
- 夜晚显示当日结算。
- 支持简化材料购买。
- 进入下一天并恢复能量。

验收标准：

- 玩家可以从第 1 天玩到第 3 天。
- 缺材料或缺能量时有明确提示。
- 金币、能量、材料变化正确。
- 玩法可用占位 UI，但流程不能依赖开发者手动操作。

## 4. Phase 2：Alpha

目标：让核心系统开始产生取舍和重复游玩价值。

必须完成：

- 加入完美复原、保留痕迹、创意再造三种修补方式。
- 加入顾客类型。
- 加入基础口碑方向。
- 加入轻量店铺升级。
- 扩展订单和材料数量。

验收标准：

- 不同修补方式会影响收入、能量、口碑或真实度。
- 不同顾客对修补结果有不同反馈。
- 店铺升级能改变至少一种资源上限或效率。

## 5. Phase 3：Beta

目标：补足叙事反馈、数值平衡和新手体验。

必须完成：

- 加入手机界面。
- 加入日记和朋友圈文本。
- 加入新手引导流程。
- 调整金币、能量、材料、真实度、口碑平衡。
- 补充基础音效、动效和视觉统一。

验收标准：

- 玩家能在不读开发文档的情况下理解主要操作。
- 夜晚反馈能反映白天选择。
- 数值压力温和但有效，不破坏治愈感。

## 6. Phase 4：内容扩展

目标：让小店形成更长期的记忆和变化。

候选内容：

- 更多顾客故事。
- 特殊订单。
- 店铺装饰变化。
- 结局方向。
- 收藏、成就、节日或主题活动。

验收标准：

- 内容扩展不破坏 MVP 闭环。
- 新系统必须服务“不完美也值得被珍惜”的主题。

## 7. 暂不做

以下内容在基建阶段不做：

- 复杂剧情。
- 多结局。
- 深度经济模拟。
- 大量正式美术资产。
- 复杂口碑权重系统。
- 完整手机社交内容。
